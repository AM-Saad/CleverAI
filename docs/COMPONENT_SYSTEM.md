# Component System

> How components are organized, named, and built. Layer above the tokens
> ([docs/DESIGN_SYSTEM.md](DESIGN_SYSTEM.md), [app/DESIGN.md](../app/DESIGN.md)).
> Inventory is generated by `yarn design:components` ‚Üí `docs/component-audit/`.

## Tiers

| Tier | Location | What it is |
|---|---|---|
| **Primitive** | `app/components/ui/` (`Ui*`) | Reusable, app-agnostic building blocks. Thin, opinionated wrappers over the themed Nuxt UI, styled only with tokens. No feature/business logic. |
| **Pattern** | `app/components/patterns/` | Cross-feature *compositions* of primitives (e.g. `UiConfirmDialog`, page section shells). Still feature-agnostic. |
| **Feature component** | `app/features/<feature>/components/` | Feature-specific UI. Composes primitives + patterns. |
| **Container** | `app/features/<feature>/components/containers/` or `containers/` | Data/logic orchestration; little markup of its own. Imported by pages. |
| **Page / Layout** | `app/pages/`, `app/layouts/` | Routes. Compose containers + feature components. |

**Canonical feature tree is `app/features/`.** The legacy `app/components/{board,review,language}/` mirror is being retired (see `docs/component-audit/duplication.md`).

## Naming & auto-import

- Primitives are **`Ui`-prefixed** PascalCase (`UiButton`, `UiModal`) ‚Äî they read clearly against Nuxt UI's `U*` and signal "design-system primitive."
- Nuxt auto-imports `app/components/**`. Feature code under `app/features/**` is imported explicitly (pages import `~/features/<f>/containers/...`).
- **Policy: feature/page/pattern code uses `Ui*`, not Nuxt UI `U*` directly.** Primitives are the only place that wraps `U*`. This keeps theming and API decisions in one layer (enforced ‚Äî see below).

## Surface taxonomy

| Primitive | Role | Rule of thumb |
|---|---|---|
| `UiCard` | Discrete content object | Use when the object itself is the content: note, workspace, stat, review item. |
| `UiPanel` | Structural region | Use when the surface organizes app UI: settings section, side pane, filter panel. |
| `UiInteractiveCard` | Clickable/selectable object, bare anatomy | Use instead of wrapping `UiCard` in a link/button. It owns keyboard and ARIA state but has no built-in content structure ‚Äî only header/default/footer slots. Use for tiles/pickers (upload dropzone, OAuth provider choice, credit pack) where each consumer lays out its own content. |
| `UiItemCard` | Structured content row/tile | Use for repeated list items that share a fixed anatomy: leading icon, kicker, title, badges, actions (board cards, word-bank rows, note rows). Prefer over `UiCard` when the row needs that anatomy for free instead of hand-rolling it each time. |
| `UiListCard` | Compact linear row | Use for simple leading+title+description+trailing rows in a list (settings-adjacent lists, switchers, material lists). Lighter than `UiItemCard` ‚Äî no kicker/badges anatomy. |
| `UiOverlaySurface` | Floating surface shell | Internal building block consumed by other overlay primitives (`DialogModal`, `TiptapEditor`, `TagInput`, sheets) for modal, drawer, popover, menu, toast, tooltip chrome. Not meant to be reached for directly by feature code ‚Äî use `UiModal`/`UiSheet`/`UiPopover`/`UiActionMenu` instead. |

Do not nest independent interactive controls inside `UiInteractiveCard`. If an object needs secondary actions, place a `UiActionMenu` beside the card or use a non-interactive `UiCard`/`UiItemCard` with explicit child controls.

## Primitive contract

Every primitive should:

1. **Wrap Nuxt UI where one exists** (`UiButton`‚Üí`UButton`, `UiModal`‚Üí`UModal`, `UiBadge`‚Üí`UBadge`, `UiFormField`‚Üí`UFormField`), inheriting the theming already set in [app/app.config.ts](../app/app.config.ts). Build standalone only when Nuxt UI has no equivalent.
2. **Declare classes with `tv()`** from [app/components/ui/variants.ts](../app/components/ui/variants.ts) ‚Äî never hand-rolled `computed` class maps. `UiCard` ([app/components/ui/UiCard.vue](../app/components/ui/UiCard.vue)) is the reference implementation.
3. **Use the shared prop vocabulary** ‚Äî reuse the canonical scales, don't invent synonyms:
   - `size`: `xs | sm | md | lg | xl` (`SIZES`)
   - `tone`: `primary | neutral | success | warning | error | info` (`TONES`) ‚Äî semantic color role
   - `variant`: visual style within the component (e.g. `solid | outline | ghost | soft`)
   - state booleans: `loading`, `disabled`
4. **Cover all states** (from the Component Checklist in [app/DESIGN.md](../app/DESIGN.md)): default, hover, focus (tokenized ring via `focusRing`), active, disabled, loading, error.
5. **Style only with tokens** ‚Äî passes `yarn design:check`. Translucency via `color-mix()` over a token.
6. **Be controllable & composable**: `v-model` where stateful, pass-through `class`, expose slots for content.

## Variant authoring (`tv()`)

```ts
import { tv, type VariantProps, TONES, focusRing } from '~/components/ui/variants'

const button = tv({
  base: `inline-flex items-center justify-center rounded-[var(--radius-lg)] ${focusRing}`,
  variants: {
    tone: { primary: 'bg-primary text-on-primary', /* ‚Ä¶ */ },
    size: { sm: 'h-8 px-3 text-sm', md: 'h-9 px-4', lg: 'h-10 px-5' },
    loading: { true: 'opacity-70 pointer-events-none' },
  },
  defaultVariants: { tone: 'primary', size: 'md' },
})
export type ButtonVariants = VariantProps<typeof button>
```

Multi-part components use `slots` (see `UiCard`). The tone ‚Üí color matrix (button/field colors + hover/active states) is owned by the Nuxt UI theme in `app/app.config.ts` ‚Äî the single source. Custom primitives that need a tone fill compose tokens directly (`bg-success/10`, `text-success-text`, ‚Ä¶).

## Enforcement

- `yarn design:components` ‚Äî inventory + duplication report (`docs/component-audit/`).
- `yarn design:check` ‚Äî token-level gate (already wired to CI + pre-commit).
- `yarn design:boundaries` ‚Äî component-boundary regression gate (wired to pre-commit). It fails on **new** raw `<button>`/`<dialog>`/`<input>`/`<select>`/`<textarea>`, ad-hoc modal overlays, hand-rolled card chrome, or direct Nuxt UI `U*` usage beyond the current `docs/component-audit/components.json` baseline. Use `design-allow` on legitimate native/editor/canvas exceptions.

## The catalog

`/design-system` (internal route) renders every primitive √ó all variants/sizes/states plus the token palettes ‚Äî the living, visible definition of the system. Keep it updated when adding a primitive.
