# Component System

> How components are organized, named, and built. Layer above the tokens
> ([docs/DESIGN_SYSTEM.md](DESIGN_SYSTEM.md), [app/DESIGN.md](../app/DESIGN.md)).
> Inventory is generated by `yarn design:components` → `docs/component-audit/`.

## Tiers

| Tier | Location | What it is |
|---|---|---|
| **Primitive** | `app/components/ui/` (`Ui*`) | Reusable, app-agnostic building blocks. Thin, opinionated wrappers over the themed Nuxt UI, styled only with tokens. No feature/business logic. |
| **Pattern** | `app/components/patterns/` | Cross-feature *compositions* of primitives (e.g. `UiConfirmDialog`, page section shells). Still feature-agnostic. |
| **Feature component** | `app/features/<feature>/components/` | Feature-specific UI. Composes primitives + patterns. |
| **Container** | `app/features/<feature>/components/containers/` or `containers/` | Data/logic orchestration; little markup of its own. Imported by pages. |
| **Page / Layout** | `app/pages/`, `app/layouts/` | Routes. Compose containers + feature components. |

**Canonical feature tree is `app/features/`.** The legacy `app/components/{board,review,language}/` mirror is being retired (see `docs/component-audit/duplication.md`).

## Naming & auto-import

- Primitives are **`Ui`-prefixed** PascalCase (`UiButton`, `UiModal`) — they read clearly against Nuxt UI's `U*` and signal "design-system primitive."
- Nuxt auto-imports `app/components/**`. Feature code under `app/features/**` is imported explicitly (pages import `~/features/<f>/containers/...`).
- **Policy: feature/page/pattern code uses `Ui*`, not Nuxt UI `U*` directly.** Primitives are the only place that wraps `U*`. This keeps theming and API decisions in one layer (enforced — see below).

## Primitive contract

Every primitive should:

1. **Wrap Nuxt UI where one exists** (`UiButton`→`UButton`, `UiModal`→`UModal`, `UiBadge`→`UBadge`, `UiFormField`→`UFormField`), inheriting the theming already set in [app/app.config.ts](../app/app.config.ts). Build standalone only when Nuxt UI has no equivalent.
2. **Declare classes with `tv()`** from [app/components/ui/variants.ts](../app/components/ui/variants.ts) — never hand-rolled `computed` class maps. `UiCard` ([app/components/ui/UiCard.vue](../app/components/ui/UiCard.vue)) is the reference implementation.
3. **Use the shared prop vocabulary** — reuse the canonical scales, don't invent synonyms:
   - `size`: `xs | sm | md | lg | xl` (`SIZES`)
   - `tone`: `primary | neutral | success | warning | error | info` (`TONES`) — semantic color role
   - `variant`: visual style within the component (e.g. `solid | outline | ghost | soft`)
   - state booleans: `loading`, `disabled`
4. **Cover all states** (from the Component Checklist in [app/DESIGN.md](../app/DESIGN.md)): default, hover, focus (tokenized ring via `focusRing`), active, disabled, loading, error.
5. **Style only with tokens** — passes `yarn design:check`. Translucency via `color-mix()` over a token.
6. **Be controllable & composable**: `v-model` where stateful, pass-through `class`, expose slots for content.

## Variant authoring (`tv()`)

```ts
import { tv, type VariantProps, TONES, focusRing } from '~/components/ui/variants'

const button = tv({
  base: `inline-flex items-center justify-center rounded-[var(--radius-lg)] ${focusRing}`,
  variants: {
    tone: { primary: 'bg-primary text-on-primary', /* … */ },
    size: { sm: 'h-8 px-3 text-sm', md: 'h-9 px-4', lg: 'h-10 px-5' },
    loading: { true: 'opacity-70 pointer-events-none' },
  },
  defaultVariants: { tone: 'primary', size: 'md' },
})
export type ButtonVariants = VariantProps<typeof button>
```

Multi-part components use `slots` (see `UiCard`). Reuse `toneText`/`toneBgSolid`/`toneBgSoft`/`toneBorder` from `variants.ts` for token-correct color sets.

## Enforcement

- `yarn design:components` — inventory + duplication report (`docs/component-audit/`).
- `yarn design:check` — token-level gate (already wired to CI + pre-commit).
- *(Planned)* a component gate failing on **new** raw `<button>`/`<dialog>`/`<input>` or ad-hoc modal overlays in app code, and on feature code importing `U*` where a `Ui*` wrapper exists — with a `design-allow` escape hatch.

## The catalog

`/design-system` (internal route) renders every primitive × all variants/sizes/states plus the token palettes — the living, visible definition of the system. Keep it updated when adding a primitive.
